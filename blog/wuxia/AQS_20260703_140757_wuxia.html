壹回 · 心法总纲
AQS 是什么来头？

AQS（AbstractQueuedSynchronizer），Java 并发包的绝对核心，Doug Lea 的封神之作。

ReentrantLock、Semaphore、CountDownLatch、ReentrantReadWriteLock……这些你天天用的同步器，全都是 AQS 的传人。

🧘 类比：AQS 就像武侠世界里的「九阳神功」——本身不是一套招式，而是一套内功心法。练成了九阳神功，你学太极拳、太极剑、乾坤大挪移都事半功倍。AQS 就是这套内功，基于它你可以写出自己的同步器。

AQS 只有两个核心：

  • state（状态值）——一个 volatile int，不同的同步器赋予它不同含义。ReentrantLock 用它表示「重入次数」，Semaphore 用它表示「剩余许可数」。
  • CLH 队列（等待队列）——获取资源失败的线程，打包成 Node 塞进一个双向队列里排队，公平得很。
说书人插话：AQS 的名字里有个 Abstract，但它不是抽象类让你直接用的——它是给写同步器的人用的。普通开发者用 ReentrantLock 就够了，但面试喜欢问 AQS，因为懂了 AQS 才算真正懂 Java 并发。

贰回 · 独门绝技
AQS 的两大门派

AQS 支持两种资源获取模式，对应两大门派：

  • 独占模式（Exclusive）——同一时刻只有一个线程能拿到资源。ReentrantLock 就是独占模式，像卫生间，一个人用，其他人排队。
  • 共享模式（Shared）——多个线程可以同时拿到资源。Semaphore 就是共享模式，像停车场，有 N 个车位就可以进 N 辆车。
  • 模板方法模式——AQS 把「入队/出队/阻塞/唤醒」的骨架写好了，子类只需实现 tryAcquire / tryRelease（独占）或 tryAcquireShared / tryReleaseShared（共享）。
  • 公平锁 vs 非公平锁——公平锁严格按 CLH 队列顺序拿锁；非公平锁允许新来的线程先抢锁，吞吐量更高，ReentrantLock 默认非公平。
说书人插话：面试官常问「公平锁和非公平锁哪个快？」——非公平锁快，因为减少了线程切换。但公平锁能避免饥饿。高并发场景下，非公平锁通常是更优选择。
⚔ 独占模式
• 同一时刻只有一个线程
• ReentrantLock
• tryAcquire/tryRelease
• 像卫生间，一人一间
🛡 共享模式
• 多个线程可以同时获取
• Semaphore / CountDownLatch
• tryAcquireShared
• 像停车场，多位可进

叁回 · 招式拆解
CLH 队列是如何运转的？

AQS 的精髓全在 CLH 队列（Craig, Landin, and Hagersten 队列的变种）。理解了这个队列，AQS 你就懂了一半。

【CLH 等待队列运行流程图】
⚡ 口诀：基于 AQS 写一个最简单的互斥锁
// 只需要继承 AQS，实现 tryAcquire / tryRelease
class MyLock extends AbstractQueuedSynchronizer {

    // 独占模式：尝试获取锁
    protected boolean tryAcquire(int acquires) {
        if (compareAndSetState(0, 1)) {
            setExclusiveOwnerThread(Thread.currentThread());
            return true;
        }
        return false;
    }

    // 独占模式：尝试释放锁
    protected boolean tryRelease(int releases) {
        setExclusiveOwnerThread(null);
        setState(0);
        return true;
    }
}

// 对外暴露的锁 API
class MyMutex implements Lock {
    private final MyLock sync = new MyLock();

    public void lock()    { sync.acquire(1); }
    public void unlock()  { sync.release(1); }
    // ... 其他方法省略
}
说书人插话：上面这段代码就是 ReentrantLock 的简化版！Doug Lea 把复杂的入队、park、唤醒逻辑全藏在 acquire() 里，子类只需关心「怎么判断能获取锁」这一件事。这就是模板方法模式的威力。

肆回 · 江湖险恶
AQS 有哪些坑？

AQS 虽强，但用错了就是走火入魔。以下是几个常见险恶之地：

坑一：tryAcquire 没考虑可重入
如果子类实现 tryAcquire 时不判断「当前线程是否已经是持有者」，那同一个线程第二次调用 lock() 会死锁自己。ReentrantLock 的做法是：如果当前线程 == exclusiveOwnerThread，state++，返回 true。

坑二：当且仅当条件用 while 循环检查
线程被唤醒后，必须重新检查条件（因为可能是虚假唤醒），所以 await 条件必须用 while 包裹，不能只用 if。

坑三：共享模式的 tryAcquireShared 返回值含义搞错
返回值 < 0 表示失败，= 0 表示获取成功但后续线程不能获取，> 0 表示获取成功且后续线程也可以获取。Semaphore 的核心就在这里，搞错就出 bug。

前辈指点：学 AQS 的最佳路径是先搞懂 ReentrantLock 的独占模式，再学 Semaphore 的共享模式，最后看 ConditionObject（AQS 的内部类，实现 wait/notify 的功能）。把这三座山头拿下，Java 并发包你就横着走了。

说书人插话：AQS 源码里有大量位运算和 CAS，第一次看会很痛苦。建议配合「手撕 AQS」的视频一起看，边看边在纸上画 CLH 队列的变化，画三遍就通了。
