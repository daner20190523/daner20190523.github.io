攻技记

—— 一个 Skill 从无到有的完整推演，三层试错，一次破局 ——

AI Skill 设计 · 2026年7月
录 · daner技术栈
壹
兵 临 城 下 —— 这张地图上还缺一个人

2026年7月的一个下午，tech-knowledge-card 已经有了五个子风格。编辑杂志风讲"技术是什么"，旧报纸风讲同样的内容换了一张皮，武侠漫画风用江湖武功比喻底层原理，探索式风钻进发明家的脑子里追问"这东西为什么被造出来"。

五个人站在地图上，各有各的位置。但当你把这张地图摊平，会看到一个醒目的空白——

"技术是什么"有人讲了。"技术为什么被发明"也有人讲了。但是"技术难题怎么被啃下来的"——没有人在讲。

不是讲 Redis 的原理，是讲 ZooKeeper 脑裂那个问题：我先试了什么？为什么不行？再试了什么？又卡在哪里了？第三次怎么突然看清本质的？

⚑ 军 情 急 报

tech-knowledge-card 覆盖了"认知层"（技术是什么）和"溯源层"（为什么发明），但缺少"实践层"——一个工程师面对真实问题时层层探索、节节败退、最终破局的全过程。已有的五个子风格，没有一个能承载"三败出真知"的叙事。

◆ 若不破此关

用户提到"攻克""攻坚""踩坑""怎么解决的"时，系统没有对应的内容类型。要么强行塞进标准武侠风（丢失了探索过程的张力），要么用 wuxia_why（视角错位——发明家和工程师看到的失败不一样）。读者想看"怎么想才能破局"，得到的却是"这个技术是干嘛的"——需求和交付物的错位，会慢慢磨掉信任。

已有的"方案"很诱人——直接让 wuxia_why 改改就能用？加一个"失败故事"板块塞进标准武侠风？或者干脆把"三败"写成 Editorial 风的 case study？每一个听起来都行，但每一个仔细推演都有硬伤。

我决定动手了。那时我还不知道，前面有三个错误答案在等我。

攻坚者旁白 "最危险的时刻不是没方案，是有好几个看起来都能用的方案——它们只是看起来能。差之毫厘，整张地图的干净度就毁了。"

贰
逐 鹿 破 阵 —— 三层试错，每一层都离答案更近

以下，是这场 Skill 设计的大事记录——不是照搬已有的 wuxia_why，不是简单加个板块，而是从需求本身逼出一个不可替代的叙事结构。

第一阵 败
思路：直接复用 wuxia_why 的探索叙事模板。反正它已经有"失败→突破"的结构了——把发明家的试错改成工程师的试错，内容换了不就行了？
沙盘推演

wuxia_why 的核心是"小孩子都能懂的发明叙事"——它的失败是概念层面的（Paxos 太难理解、Multi-Paxos 缺少工程化规范）。如果强行改成工程师视角，就会出现一个致命矛盾：专业度要求和"小孩能懂"的语言风格无法共存。真实的工程师失败涉及锁竞争、网络分区、存储引擎——这些只能用术语讲。

为什么败了 发明家的失败和工程师的失败是两类信息。发明家的失败可以从论文里找（外部、历史），工程师的失败必须从推演里酿（内部、实时）。wuxia_why 的结构是为"外部叙事"优化的——读论文的人不需要知道 10 万 QPS 下锁竞争的精确数据，但做攻坚手札的人需要。硬套模板的结果是：叙事结构不匹配内容密度——要么专业度缩水（失去了本技能的核心辨识度），要么语言炸穿"小孩能懂"的预设。
悟到的 "wuxia_why 和 conquest 的根本区别不是内容，是视角的焦距。wuxia_why 用望远镜看（概念级），conquest 要用显微镜看（细节级）。同一个叙事骨架撑不住两种焦距——需要独立的骨架。"
第二阵 败
思路：不新建子技能，在标准武侠风 SKILL_wuxia 里加一个"踩坑实录"板块。用户说"踩坑"的时候触发标准武侠，但在末尾追加一个失败故事区。
沙盘推演

标准武侠风是"四维分析"骨架（是什么→为什么用→怎么用→有什么问题）。"踩坑"是"怎么用"下的一个子板块。但用户的要求是：探索过程本身就是文章的主线，不是正文的附录。

为什么败了 在主流的"认知型"卡片里塞一个"探索型"板块，就像在百科条目末尾加一篇日记——形式上的拼接不等于结构上的融合。读者先看完"Redis 是什么、怎么用"的标准四段，突然转到"但我们在这个场景下试了三种方案都不行"——注意力弧断了。探索的张力需要从头到尾的一条脊柱，不能是拼贴。
悟到的 "不是所有的内容都适合'add-on'。当一种叙事需要的时间弧长和情绪曲线与已有模板完全不同时，它必须是一个独立的结构——不是第四种风格，是第六种视角。"
第三阵 败
思路：那就新建一个子技能 + 独立模板。但内容怎么写？按"Editorial 风 + 案例叙事"来——把三败当作 Case Study，用杂志的排版讲。
沙盘推演

用户的需求里有"依旧是武侠风 HTML 排版"。但 Editorial 是另一种视觉体系（衬线体+版式留白+彩色装饰条），和武侠基色不在一个体系里。而且"大事记"这个格式需要时间轴——左侧竖线、菱形节点、阵号标记——这些组件在 Editorial 模板里全都没有。

为什么败了 真正的矛盾不是"要不要新建模板"，而是"新模板的视觉基因应该从哪继承"。Editorial 的排版语言是为"杂志阅读"优化的（静态、单向、信息密度高），但攻坚手札需要"编年体时间轴"的视觉节奏——动态、因果、一步步推进。内容和视觉的节奏不匹配，等于穿错了衣服。
悟到的 "视觉体系必须和叙事节奏同频。攻坚手札的底层节奏是时间线性 + 因果递进——每败一次离真相近一步。这决定了排版必须是年鉴式的：竖线时间轴是主视觉，节点标记是节奏点，四回目是章节锚。Editorial 的多列版式不行，Newspaper 的栏栅也不行，只有武侠宣纸卷轴能撑起这条时间线——但组件全部要从头造。"
攻坚者旁白 "连败三阵的时候，其实心里已经有答案了——问题不在"做不做"，在"怎么做才不会被已有的五个人盖住"。第一阵告诉我视角不能复用，第二阵告诉我结构不能拼接，第三阵告诉我视觉不能平移。三张反面教材摊开来，正面答案的轮廓就已经出来了。"

一 剑 封 喉 —— 四回目编年体：把叙事翻译成 CSS
决胜局 成
思路：从用户的原始需求里找结构——"发现问题→1个想法不行→第2个不行→第3个不行→层层递进→最终成功→但存在新问题"。这句话本身就是四回目的骨架。把它拆开：兵临城下（发现问题）→ 逐鹿破阵（三战三败）→ 一剑封喉（突破）→ 余波未平（新问题）。直接继承武侠宣纸基色，但重新设计 19 个专属 CSS 组件来承载编年体叙事。
★★★ 破阵时刻 ★★★
四回目编年体，通
不是"用了什么技术框架"的突破，而是"找到了不可替代的叙事视角"的突破——编年体大事记格式 + 因果递进结构 + 专业度不减 = 第六个人的生态位。
「 前三阵不是白败的，它们帮我画出了地图上那张唯一的空椅子 」
  • 利器一 · 叙事骨架直取需求原句 用户说"1个想法不行，第2个不行，第3个不行"——这句话本身就是四回目的风骨。没有用任何现成的叙事模板，而是把用户的需求当成结构来读。"三败"对应贰回的大事记时间线，"新问题"对应肆回的余波未平。
  • 利器二 · 视觉组件与叙事单元一一映射 每个叙事单元一个独立的 CSS 组件：军情急报 → .crisis-scene（朱砂左边框），大事时间线 → .chronicle-log（竖线+菱形节点），每阵四层 → .be-attempt + .sandbox-box + .be-failure + .be-lesson，破阵时刻 → .breakthrough-box（金边+光晕），新问题 → .new-threat（朱砂警告框）。共 19 个专属组件，全部继承武侠宣纸基色。
  • 利器三 · 关键词分发零冲突 "攻克""攻坚""踩坑""手札""破局"这些关键词和已有的 wuxia_why（"发明""创造""想出来"）天然不重叠——一个是"怎么啃下来的"（事后回忆），一个是"怎么被造出来的"（溯源叙事）。两个视角的用户在使用场景上就不会混淆。

回头看，这道题真正的解法不是某一招，而是先画地图、再找空椅子。六个子风格中，五个站好了自己的位置——conquest 不是"更好的第五个"，而是"少了就空着的第六把椅子"。

★ ☆ ★
肆
余 波 未 平 —— 城建起来了，但新问题不会停下

SKILL_wuxia_conquest 落地了。684 行 HTML，19 个 CSS 组件，三败成稿。但真实工程的魅力恰恰在于——破城之后，新狼烟从不会缺席。

☡ ⚠ 新狼烟 · 其一：内容质量的天花板

攻坚手札要求"专业度不减"——每一次失败必须有具体的技术细节和因果链条。这在批量生成场景下是硬约束：AI 需要足够深的技术上下文才能写出有说服力的"三败"。如果只是一个泛泛的技术名词（比如"攻克高并发"），三败会空洞。内容质量的底线比认知型卡片高一个数量级。

☡ ⚠ 新狼烟 · 其二：与 wuxia_why 的长尾边界模糊

虽然"攻克"和"发明"在关键词层面天然分清，但当用户说"讲讲 Redis 怎么被造出来的，以及早期踩过的坑"——这就同时触发两个视角。目前的策略是"命中谁就进谁"，但如果用户意图是混合型，可能会期待两个视角的融合，而非二选一。

☡ ⚠ 新狼烟 · 其三：模板复杂度 vs 维护成本

19 个 CSS 组件意味着每一次模板微调都可能产生涟漪效应。如果武侠基色（--cinnabar、--gold、--paper）在未来被调整，conquest 模板需要同时验证——独立的视觉组件越多，耦合的测试面越广。

◆ 下一城

这场攻坚教给我的不只是一个 Skill 的诞生方法。它教给我的是——设计一个新能力时，第一件事不是写代码，是画一张"已有体系的地图"、找到那张空着的椅子，然后从用户需求本身逼出一个不可替代的叙事结构。下次遇到同样棘手的 Skill 设计题，这套"先画地图→再找空位→三败出真知"的路子还能再用。

攻坚者旁白 "创造一个 Skill 和解决一个技术问题没什么不同——关键在于你能不能看清：问题的本质不在你要造的这件东西本身，而在它周围已经站了谁。这张地图，画对了，答案自然就有了。"
