壹回 · 心法总纲
集齐令是什么来头？

话说并发江湖，有一门绝学叫 CountDownLatch——江湖人称「集齐令」。

它的心法只有一个：等有 N 个线程都报到，主线程才能继续走。

🏯 类比：就像公司门禁——早晨要有 5 个部门的人全部到齐，保安才开门放行。每个部门来一个人，计数器减 1，减到 0 门才开。来早的部门在门口等着，来晚的拖大家后腿。

集齐令继承自 AQS（AbstractQueuedSynchronizer）的门派，内部用一个 state 字段当计数器。初始化时设定 state = N，每调用一次 countDown()，state 减 1；state 变成 0 时，所有在 await() 上等待的线程被唤醒。

说书人插话：AQS 是并发江湖的「内功心法」，ReentrantLock、Semaphore、CountDownLatch 全是它的传人。学会了 AQS，一半 Java 并发包你就通了。

贰回 · 独门绝技
为何要练这门绝学？

集齐令这门功夫，专治「等所有人干完才能继续」的场景，有四大绝技：

  • 一次性的闸刀——计数器归零后不能重置，适合「一次性开关」场景，用完即弃，干净利落。
  • await 可超时——等不及可以设超时，避免无限阻塞，江湖险恶要有退路。
  • 轻量无锁竞争——底层 CAS 操作，性能比 Thread.join() 循环等待高得多。
  • 支持多等一 & 一等多——N 个线程等 1 个主线程（开始信号），或 1 个主线程等 N 个 worker（完成信号）。
说书人插话：有人问「CyclicBarrier 和 CountDownLatch 有啥区别？」——集齐令是一次性的闸刀，用完报废；CyclicBarrier 是可重用的，像健身房团课，每节课结束可以重新开始下一节。
⚔ Thread.join()
• 只能等单个线程
• 阻塞主线程
• 无法超时控制
• 代码啰嗦
🛡 CountDownLatch
• 同时等 N 个线程
• 支持超时 await
• 优雅的计数器语义
• 并发包标准方案

叁回 · 招式拆解
如何修炼这门绝学？

修炼集齐令，只需三步：初始化 → countDown → await。

【招式运行流程图】
☯ 口诀一：主线程等待 N 个 worker 完成
// 集齐令初始化：计数器 = 3（等 3 个 worker）
CountDownLatch latch = new CountDownLatch(3);

// 启动 3 个 worker 线程
for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        // 干完活，计数器减 1
        doWork();
        latch.countDown();
    }).start();
}

// 主线程等待，计数器归零才继续
latch.await();
System.out.println("所有 worker 干完了，主线程继续！");
⚡ 口诀二：带超时的 await（防止永远等下去）
// 最多等 5 秒，超时返回 false
boolean done = latch.await(5, TimeUnit.SECONDS);
if (!done) {
    System.out.println("超时了！有 worker 没干完，做降级处理");
}
说书人插话：很多初学者忘了处理 await 的超时返回值，线上出了问题线程一直卡着，这是大忌！永远要检查返回值，做超时降级。

肆回 · 江湖险恶
这门绝学有哪些坑？

集齐令虽好，但有几处险恶之地，不注意就会走火入魔：

坑一：countDown() 没放到 finally 块
如果 worker 线程抛出异常，countDown() 没执行，计数器永远到不了 0，主线程永远 await 不下去——死等！正确做法：把 countDown() 放在 finally 块里，确保无论如何都会减 1。

坑二：await 超时没做处理
await(timeout) 返回 false 时，说明有线程没干完。如果忽略返回值，主线程继续走，数据可能不完整。正确做法：检查返回值，做降级或告警处理。

坑三：重用 Latch
CountDownLatch 是一次性的，计数器归零后不能重置。如果业务需要重复等待，用 CyclicBarrier 或重新 new 一个 Latch。强行重用会出诡异问题。

前辈指点：CountDownLatch 最适合「主线程等 worker 全部完成」或「worker 等主线程发开始信号」两种场景。如果是多个阶段的等待（阶段一完成→阶段二开始），考虑用 Phaser，它是集齐令的进阶版，支持动态注册和多个阶段。
